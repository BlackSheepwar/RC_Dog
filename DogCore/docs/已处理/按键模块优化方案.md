# 按键模块优化方案

> 对现有按键架构（BSP_GPIO → Debounce → APP_Key → APP_Key_Cmd → Task）的一次专项精简与重构建议。

---

## 一、现状分析

### 当前模块依赖链

```
BSP_GPIO (bsp_gpio)
    ↓  读电平
Debounce (middleware)   ← 按键专用防抖 + 状态机
    ↓  事件
APP_Key (app_key)
    ↓  消息队列
Task_PAR (task_par)
    ↓  分发
APP_Key_Cmd (app_key_cmd)
```

### 各模块职责与问题

| 模块 | 当前职责 | 问题 |
|---|---|---|
| `Middlewares/Debounce/` | 按键消抖 + 按键状态机 + 按键事件生成 | **按键特化**，无其他消费者，不该在 Middlewares 层 |
| `APP/APP_Key/app_key.c` | 管理按键池，调用 Debounce_Update，入队 | 只做胶水工作，实质逻辑全在 Debounce 中 |
| `APP/APP_Key/app_key_cfg.h` | 按键注册参数表 | 结构合理，保持 |
| `APP/APP_Key/app_key_cmd.c` | 事件 → 处理函数分发 | 含废弃注释代码 |
| `TASK/task_key.c` | 初始化 + 周期扫描 | 结构合理 |
| `TASK/task_par.c` | 接收队列 → 分发命令 | 名字含糊（par?），类型依赖 debounce.h |
| `Core/Src/freertos.c` | 包含 debounce.h 建队列 | 被迫暴露 Middleware 细节 |

### 依赖关系验证

```bash
$ grep -r "debounce.h"
Core/Src/freertos.c:28:      #include "debounce.h"    # 为了 Debounce_Event_packet_t
User/APP/APP_Key/app_key.h:18:#include "debounce.h"    # 为了 Debounce_Event_t / Debounce_Key_t
User/APP/APP_Key/app_key_cmd.h:14:#include "debounce.h"  # 为了 Debounce_Event_t
User/Middlewares/Debounce/debounce.c:12:#include "debounce.h"
User/Middlewares/Debounce/debounce.h:2:/* @file debounce.h */
```

所有外部引用都来自按键模块。**没有其他模块（CAN、舵机、步态等）使用 Debounce**。结论明确：Debounce 是按键专属，理应属于 APP_Key。

---

## 二、核心优化：合并 Debounce → APP_Key

### 2.1 方案

将 `debounce.c/h` 的代码完整移入 `app_key.c/h`：

- `Debounce_Key_t` → `App_Key_t`（按键实例结构体）
- `Debounce_Event_t` → `App_Key_Event_t`（事件枚举）
- `Debounce_Event_packet_t` → `App_Key_EventPacket_t`（消息结构体）
- `Debounce_State_t` → `App_Key_State_t`（内部状态枚举，可对外隐藏）
- `Debounce_Update()` → `App_Key_DebounceUpdate()`（静态函数，不暴露）

### 2.2 删除文件

- `User/Middlewares/Debounce/debounce.c`
- `User/Middlewares/Debounce/debounce.h`

### 2.3 受影响的文件及改动

| 文件 | 改动 |
|---|---|
| `app_key.h` | 吸收 Debounce 的类型定义 + 事件枚举 + 消息结构体 |
| `app_key.c` | 吸收 `Debounce_Update()` 状态机逻辑，作为静态函数 |
| `app_key_cmd.h` | `#include "debounce.h"` → `#include "app_key.h"` |
| `task_par.c` | `Debounce_Event_packet_t` → `App_Key_EventPacket_t`；更新 include |
| `freertos.c` | `#include "debounce.h"` → `#include "app_key.h"`（消息队列创建处） |
| `main.h` | 检查 `extern` 声明是否需要更新类型名 |

### 2.4 收益

- **减少一层目录**：消除 Middlewares/Debounce，模块边界更清晰
- **减少一个抽象层**：APP_Key 不再只是胶水，而是完整的功能模块
- **消除类型命名混乱**：`App_Key_*` 统一前缀，`Debounce_*` 灭迹
- **简化 include 链**：app_key_cmd.h 和 freertos.c 不再需要间接包含

## 三、次要优化建议

### 3.1 DEBOUNCE_TICKS 改为可配置参数

当前 `DEBOUNCE_TICKS = 2` 硬编码在 debounce.h。在 10ms 扫描周期下等效 20ms 防抖。

**建议**：将其下沉到每个按键的注册参数中，放入 `key_register_t` 配置表：

```c
typedef struct {
    uint8_t  key_id;
    uint8_t  long_press;     /* 长按判定阈值（单位:任务周期） */
    uint8_t  active_level;   /* 按键激活电平 */
    uint8_t  debounce_ticks; /* 防抖稳定计数（单位:任务周期） */
} key_register_t;
```

这样不同按键可以使用不同的防抖强度（例如：机械按键 3 tick = 30ms，触摸按键 1 tick = 10ms）。

**改动范围**：`app_key_cfg.h` 配置表、`App_Key_Register()` 签名、`App_Key_DebounceUpdate()` 读取 `key->debounce_ticks` 代替全局宏。

### 3.2 资源池大小专用化

当前 `app_key_pool[GPIO_MAX_NUM]`（20 个槽位），每个 `Debounce_Key_t` 约 5 字节，共 100 字节。

**建议**：用独立宏 `KEY_MAX_NUM` 代替，取值实际按键数（如 4 或 8），减少内存浪费。

```c
/* app_key.h */
#define KEY_MAX_NUM     8    /* 实际最大支持的按键数 */
```

### 3.3 队列满处理（TODO）

`app_key.c` 的队列发送函数：

```c
if (osMessageQueuePut(KEY_QHandle, &msg, 0, 0) != osOK)
{
    /* TODO: 队列满处理 */
}
```

**建议**：至少打错误计数，便于运行时诊断：

```c
static uint32_t key_queue_drop_count = 0;

if (osMessageQueuePut(KEY_QHandle, &msg, 0, 0) != osOK)
    key_queue_drop_count++;
```

错误计数器可通过调试接口（USART 命令 / LED 闪烁模式）暴露。

### 3.4 废弃注释代码清理

`app_key_cmd.c` 中包含一段被注释掉的旧代码（`Key1_OnDown` 的四腿版本，约 30 行），Git 历史中可找回。

**建议**：删除注释块。需要时从 Git 中恢复。

### 3.5 Key1_OnClick 空函数

`app_key_cmd.c` 中：

```c
static void Key1_OnClick(void) { }
```

**建议**：
- 如果确实暂时不需要，加注释说明用途或移除 TABLE 中的对应行。
- 如果保留占位，显式标注 `/* 预留：目前无操作 */`。

### 3.6 Key1_OnLongUp 语义疑点

```c
static void Key1_OnLongUp(void)
{
    APP_Servo_SetTarget(1, APP_Servo_GetCurrent(1));
    APP_Servo_SetTarget(2, APP_Servo_GetCurrent(2));
}
```

用当前值再次 SetTarget 等效于无操作。如果意图是"停止舵机运动"或"保持当前位置"，建议明确语义：

- 如果是要急停：改为 `APP_Servo_Stop(id)`（若存在）或注释说明
- 如果是复位操作：直接写具体目标角度

### 3.7 task_par.c 命名与职责

`task_par.c` 名字不明确（par 令人联想到 parameter/parallel），实际只做按键事件分发。

**建议**：
- 短期：加文件注释明确此任务只处理按键事件
- 长期：重命名为 `task_key_dispatch.c`，与 `task_key.c`（扫描任务）形成对称命名

如果按键事件处理逻辑轻量，甚至可以考虑将消息接收合并进 `task_key.c`，用一个 Task 完成扫描+分发（去掉消息队列的间接跳转）。这取决于是否需要队列解耦（例如将来按键事件可能来自 CAN/USART）。

## 四、改造后架构

```
BSP_GPIO (bsp_gpio)
    ↓  读电平
APP_Key (app_key)
  ├ 消抖 + 状态机（原 Debounce，internal）
  ├ 按键池管理
  └ 事件入队
    ↓ 消息队列
Task_PAR / Task_Key_Dispatch
    ↓  分发
APP_Key_Cmd (app_key_cmd)
  └ 配置表 → handler()
```

文件变更清单：

```
删除：
  User/Middlewares/Debounce/debounce.c
  User/Middlewares/Debounce/debounce.h

修改：
  User/APP/APP_Key/app_key.c        ← 吸收 Debounce_Update 逻辑
  User/APP/APP_Key/app_key.h        ← 吸收类型定义，改 API 签名
  User/APP/APP_Key/app_key_cfg.h    ← 增加 debounce_ticks 字段（可选）
  User/APP/APP_Key/app_key_cmd.h    ← 改 include
  User/APP/APP_Key/app_key_cmd.c    ← 清理注释代码 + 空函数处理
  User/TASK/task_par.c              ← 改类型名 + include
  Core/Src/freertos.c               ← 改 include
```

## 五、实施顺序

1. **Step 1** — 合并 Debounce → APP_Key（类型重命名 + 逻辑内聚）
2. **Step 2** — 更新所有引用点（task_par.c, freertos.c, app_key_cmd.h）
3. **Step 3** — 删除 Middlewares/Debounce 目录
4. **Step 4** — 次要优化：DEBOUNCE_TICKS 可配置、池大小专用、队列满计数
5. **Step 5** — 清理：废弃注释、空函数、语义疑点
6. **Step 6** — 可选：task_par.c 重命名 / 合并

每步完成后编译验证，避免一次改太多导致定位困难。
